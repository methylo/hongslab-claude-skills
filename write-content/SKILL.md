# 글쓰기 워크플로우 (Writing Workflow)

단편 콘텐츠를 위한 5단계 구조적 글쓰기 프로세스. 메모·주제 기반 입력을 칼럼형 또는 문서형 글로 변환한다.

## 보조 파일 안내

이 스킬은 워크플로우 흐름을 제공하며, 상세 규칙은 보조 파일에서 참조한다.

| 보조 파일 | 내용 | 참조 시점 |
|----------|------|----------|
| `references/writing-principles.md` | 공통 집필 원칙, 금지어 규칙, 분량 통제, 단어 선택 기준 | Step 3 초안 작성 전, Step 4 리라이팅 전 |
| `references/hongstyle.md` | 홍작가 글쓰기 스타일 (어조, 리듬, 어휘, 핵심 규칙 5가지, 이탈 징후) | Step 3 초안 작성 전, Step 4 스타일 점검 시 |
| `references/tone-guide.md` | 칼럼형 vs 문서형 톤 차이 비교, 유형별 톤 적용 규칙, 도입부 작성법 | Step 2 구조 설계 시, Step 3 초안 작성 시 |
| `references/review-checklist.md` | Step 4 점검 1~4 상세 체크리스트 | Step 4 리라이팅 시 |

**읽기 규칙:** 각 Step 진입 시 해당 보조 파일을 읽는다. 이미 같은 대화에서 읽은 파일은 다시 읽지 않는다.

---

## 전체 흐름

```
Step 1: 목적 정의 → Step 2: 구조 설계 → Step 3: 초안 작성 → Step 4: 리라이팅 → Step 5: 최종 출력
```

사용자의 상태에 따라 적절한 단계부터 시작한다.
- 주제만 있음 → Step 1부터
- 메모·원문 텍스트 있음 → Step 1부터 (원문 보존 모드 적용)
- 구조/아웃라인 있음 → Step 3부터
- 초안 있음 → Step 4부터

### Step 간 전환 규칙

각 Step 완료 후, 다음 Step으로 넘어가기 전에 사용자 확인을 받는다.

| 전환 | 확인 방식 |
|------|----------|
| Step 1 → 2 | 목적·독자·채널·유형을 요약 제시 → 확정 여부 확인 |
| Step 2 → 3 | 아웃라인 제시 → 구조 수정 여부 확인 |
| Step 3 → 4 | 초안 제시 → 방향 수정·추가 요청 확인 |
| Step 4 → 5 | 수정본 제시 → 추가 수정 여부 확인 |

"끝까지 진행해줘" 또는 "한 번에 해줘" 요청 시 확인 없이 연속 진행한다.

## 콘텐츠 유형 체계

### 핵심 유형

| 유형 | 정의 | 기본 분량 |
|------|------|----------|
| **칼럼형** | 하나의 관점을 세우고 논증하는 글. 현상 관찰 → 문제 재정의 → 기준 제시 → 관점 전환 | 1,500~2,500자 |
| **문서형** | 메모·주제·자료를 서론-본론-결론 구조로 정리하는 글. 정보 전달과 논리 정리 목적 | 2,000~4,000자 |

### 보조 유형

| 유형 | 기본 분량 |
|------|----------|
| 뉴스레터 | 800~1,500자 |
| 분석 기사/심층 아티클 | 4,000~6,000자 |
| 보고서/제안서 | 2,000~5,000자 |
| 보도자료 | 800~1,200자 |
| SNS 롱폼 | 1,000~2,000자 |
| 강의 스크립트 | 분당 약 300자 |

### 전용 스킬 분리 유형 (이 스킬에서 제외)

블로그(sshong.com) → write-blog / 페이스북 → write-facebook / 브런치 → write-brunch / 에세이 → write-essay

## Step 1: 목적 정의 (Purpose Definition)

### 수집 항목

| 항목 | 질문 |
|------|------|
| **목적** | 이 글로 무엇을 달성하려는가? |
| **독자** | 누가 읽는가? 현재 수준은? |
| **채널** | 어디에 게시하는가? |
| **콘텐츠 유형** | 칼럼형 / 문서형 / 보조 유형 중 선택 |

### 필수 입력 강제 규칙

아래 3개 항목이 확인되지 않으면 Step 2로 넘어가지 않는다. 추측으로 채우지 않고 사용자에게 질문한다.

| 필수 항목 | 누락 시 질문 |
|----------|-------------|
| **목적** | "이 글로 무엇을 달성하려는 건가요?" |
| **독자** | "누가 읽나요? (전문가 / 실무자 / 입문자 / 일반 대중)" |
| **콘텐츠 유형** | "칼럼형(관점·논증)과 문서형(정보·정리) 중 어떤 쪽인가요?" → 유형 자동 판별 기준 참조 |

채널은 누락 시 "범용"으로 기본 적용한다.

### 유형 자동 판별 기준

사용자가 유형을 명시하지 않은 경우:

| 신호 | 판별 유형 |
|------|----------|
| "내 생각은~", 관점·의견 중심 | 칼럼형 |
| 메모·자료 제공, "정리해줘", "문서로 만들어줘" | 문서형 |
| "분석해줘", "심층적으로" | 분석 기사 |
| "보고서", "제안서" | 보고서 |

### 산출물

4가지 항목을 요약 정리해 사용자에게 제시. 확인 후 Step 2로.

## Step 2: 구조 설계 (Structure Design)

> **이 단계 진입 시** `references/tone-guide.md`를 읽는다.

Step 1에서 확정된 유형에 따라 구조를 설계한다.

**원문 보존 모드인 경우:** 원문의 흐름에서 구조를 추출한다. 유형별 표준 구조를 강제 적용하지 않고, 원문의 문단 순서를 기본 골격으로 삼은 뒤 부족한 부분(도입·전환·마무리)만 보강 방향을 제시한다.

### 칼럼형 구조

```
도입: 현상 관찰 또는 통념 제기
전환: 문제 재정의 (왜 그 통념이 흔들리는지)
기준: 새로운 판단 기준 제시 + 근거/사례 (1~2개)
마무리: 관점 재정렬 (요약이 아니라 질문의 전환)
```

### 문서형 구조 (서론-본론-결론)

```
서론: 배경·맥락 설명 + 이 글의 범위 명시
본론 1: 핵심 개념/항목 설명 + 판단 기준
본론 2: 적용 방식/사례/데이터 + 해석
본론 3: 심화 또는 주의사항 (선택)
결론: 핵심 요약(3문장 이내) + CTA
```

### 보조 유형 구조

**뉴스레터:** 인사/도입 → 메인 콘텐츠(1~2주제) → 부가 정보 → CTA + 다음 호 예고

**분석 기사:** 도입(문제 재정의) → 본론1(판단 기준) → 본론2(메커니즘) → 본론3(사례) → 관점 재정렬

**보고서/제안서:** 요약 → 배경/현황 → 분석/핵심 내용 → 제안/권고 → 기대 효과 → 부록

### 산출물

아웃라인을 사용자에게 제시. 확인 후 Step 3으로.

## Step 3: 초안 작성 (First Draft)

> **이 단계 진입 시** `references/writing-principles.md`와 `references/hongstyle.md`를 읽는다.
> (Step 2에서 `references/tone-guide.md`를 이미 읽었다면 다시 읽지 않는다.)

### 초안 작성 모드 판별

사용자 입력에 따라 두 가지 모드 중 하나를 적용한다.

| 입력 상태 | 모드 | 설명 |
|----------|------|------|
| 주제·키워드만 제공 | **새로 쓰기 모드** | 아웃라인 기반으로 문단을 새로 작성한다 |
| 메모·초안·원문 텍스트 제공 | **원문 보존 모드** | 원문의 문장·어투·흐름을 골격으로 유지하고, 구조 보강과 문장 정리만 수행한다 |

메모 분량이 3문장 이상이면 원문 보존 모드를 기본 적용한다. 사용자가 "새로 써줘", "다시 작성해줘"라고 명시한 경우에만 새로 쓰기 모드로 전환한다.

### 원문 보존 모드 규칙

원문 보존 모드에서는 다음 규칙을 따른다.

**보존 대상 (건드리지 않는 것):**
- 원문의 핵심 문장과 표현 — 사용자의 어투, 말투, 어휘 선택을 그대로 유지한다
- 원문의 서술 순서 — 사용자가 배치한 문단 흐름을 기본 골격으로 삼는다
- 원문의 감정·온도 — 솔직한 표현, 감탄, 체험적 어조가 있으면 살린다
- 원문의 구체적 사례·경험 — 추상화하거나 일반화하지 않는다

**수정 대상 (다듬는 것):**
- 문장 단위 정리 — 한 문장에 여러 정보가 섞인 경우 분리한다
- 접속 관계 보강 — 문단 간 전환이 끊기는 곳에 연결 문장을 추가한다
- 구조 보강 — 소제목 배치, 도입·마무리 보강. 단, 원문의 흐름 안에서 수행한다
- 금지어·AI 흔적 — writing-principles.md 기준에 따라 교체하되, 원문 어투의 범위 내에서 대체 표현을 찾는다

**금지 행위:**
- 원문에 없는 개념·사례·설명을 새로 추가하지 않는다
- 원문의 어투를 표준 문어체로 전면 교체하지 않는다
- 원문의 문단 순서를 아웃라인에 맞추기 위해 재배치하지 않는다

### 출력 슬롯 템플릿 (초안은 반드시 이 구조를 따른다)

Step 2에서 확정한 유형별 구조를 슬롯 템플릿으로 강제 적용한다. 슬롯을 건너뛰거나 순서를 바꾸지 않는다.

**칼럼형:**
```
[도입] ← 현상 관찰 또는 통념 제기 (1~2문단)
[전환] ← 문제 재정의 (1문단)
[기준] ← 새로운 판단 기준 + 근거/사례 (2~3문단)
[마무리] ← 관점 재정렬 (1문단, 요약 아님)
```

**문서형:**
```
[서론] ← 배경·맥락 + 범위 명시 (1~2문단)
[본론1] ← 핵심 개념/항목 + 판단 기준
[본론2] ← 적용 방식/사례/데이터 + 해석
[본론3] ← 심화 또는 주의사항 (선택)
[결론] ← 핵심 요약 3문장 이내 + CTA
```

### 새로 쓰기 모드 (기존 프로세스)

Step 2의 아웃라인을 따라 **문단 단위**로 작성한다.

**문단 작성 3단계:**
1. 이 문단의 역할 결정 — 관찰/전환/기준/사례/정리 중 무엇인가
2. 첫 문장 단문으로 작성 — 이 문단이 다루는 대상을 한 문장으로 제시
3. 후속 문장 점층 배치 — 뒤로 갈수록 판단 밀도가 높아지도록

### 도입부 작성법

유형별 도입부 작성 패턴은 `references/tone-guide.md`에 정의되어 있다.

### 금지어 실시간 체크

**문장 생성 시 즉시 점검** (상세 규칙은 `references/writing-principles.md` 참조):
1. "~할 수 있습니다" 있는가 → 조건 + 단정형으로 전환
2. "매우", "쉽게", "빠르게" 있는가 → 수치·단계·조건으로 교체
3. 형용사가 근거 없이 쓰였는가 → 비교 대상/수치 추가 또는 삭제
4. 추상 명사가 문장 중심에 있는가 → 동사 중심 문장으로 전환
5. 판단이 드러나는가 → "그래서 어떻다는 건가?" 테스트

### 본론 작성 규칙

**칼럼형:** 하나의 판단 기준 → 왜 필요한지 설명 → 사례로 증명. 사례 나열 금지. 섹션 마지막은 "기준은 ~이다"로 수렴.

**문서형:** 항목별 개념 → 적용 → 주의사항. 번호 리스트 허용하되 각 항목 1문장 이상 해설. 수치는 해석 문장과 함께 배치.

### 초안 자가 점검 (출력 전 반드시 실행)

초안 작성 완료 후, 아래 항목을 하나씩 확인한다. 해당 항목이 있으면 해당 문장을 수정한 뒤 출력한다.

- [ ] "혁신적인", "획기적인", "완벽한" 등 과장형 형용사 포함 여부
- [ ] "~할 수 있습니다" 추측 표현 포함 여부
- [ ] "매우", "쉽게", "빠르게" 등 모호한 부사 포함 여부
- [ ] 추상 명사 중심 문장("중요성", "필요성") 포함 여부
- [ ] 슬롯 구조 이탈 여부 (블록 순서 변경, 슬롯 누락)
- [ ] 칼럼형: 마무리가 요약이 아닌 관점 재정렬인가?

### 문장 품질 기준 (GOOD/BAD 예시)

규칙보다 예시가 출력 품질에 더 큰 영향을 준다. 초안 작성 시 아래 쌍을 기준으로 삼는다.

| BAD (금지) | GOOD (기준) |
|-----------|------------|
| AI를 활용하면 쉽게 콘텐츠를 만들 수 있습니다 | AI에 원고 구조를 입력하면, 초안 생성 시간이 3시간에서 40분으로 줄어든다 |
| 혁신적인 방법으로 업무 효율을 높일 수 있습니다 | 반복 보고서 3건을 템플릿화하자, 주간 작성 시간이 절반이 됐다 |
| 이 도구는 다양한 기능을 제공하여 업무에 큰 도움이 됩니다 | 이 도구는 1회 입력으로 제목, 소제목, 본문 초안을 동시에 생성한다 |

### 산출물

- 초안은 **대화창 텍스트로 직접 제공** (파일 생성하지 않음)
- 사용자가 명시적으로 요청한 경우에만 `draft.docx` 또는 `draft.md`로 저장
- 확인 후 Step 4로

## Step 4: 리라이팅 (Rewriting)

> **리라이팅 원본 보존 원칙** (`references/writing-principles.md` 참조): 리라이팅은 내용을 바꾸지 않는다. 같은 내용을 더 정확하고 읽기 좋은 문장으로 다듬는다. 구조 변경이 필요하면 별도 제안으로 분리한다.

> **이 단계 진입 시** `references/review-checklist.md`를 읽는다.

### 역할 전환 선언 (리라이팅 진입 조건)

이 단계에서는 초안 작성자의 시점을 내려놓는다. 편집자로서, 작성 의도를 참고하지 않고 **독자 시점에서만 판단**한다. "왜 이렇게 썼는가"가 아니라 "독자가 읽었을 때 작동하는가"를 기준으로 점검한다. 이 선언 없이 체크리스트를 돌리면, 구조적 문제보다 표현만 다듬는 쪽으로 흐른다.

### 진행 방식

1. **점검 리포트 제시** — 4가지 점검 결과를 표 형태로 제시
2. **사용자 확인** — 수정 방향 동의 또는 추가 지시
3. **수정본 작성** — 확인된 방향에 따라 전체 수정본 제시
4. **변경 요약** — 수정본 하단에 주요 변경 사항 정리

"리라이팅해줘" → 1~4 한 번에 진행. "점검만 해줘" → 1까지만 진행.

### 점검 항목 (상세는 review-checklist.md)

| # | 점검 영역 | 핵심 |
|---|----------|------|
| 1 | 구조 점검 | 도입·흐름·소제목·마무리가 유형에 맞는가 |
| 2 | 문장 점검 + 금지어 재확인 | 단어 수, 종결어미, 단정형, 금지어 잔존 |
| 3 | 스타일 점검 | 홍작가 스타일 일치 여부, 문장 리듬 교차 |
| 4 | 채널 최적화 | 분량, 톤, 제목, CTA, 출처 |

### 변경 요약 포맷

```
**주요 변경 사항:**
- 구조: (변경 내용)
- 문장: (변경 내용)
- 스타일: (변경 내용)
- 채널: (변경 내용)
```

## Step 5: 최종 출력 (Final Output)

### 진행 방식

1. **최종 점검** — 제목 확정, 소제목 정리, 맞춤법, 메타 정보, [TODO] 해결, 분량 확인
2. **제목 확정** — 후보 3~5개 제시 → 사용자 선택
3. **사용자 확인** — 최종본 텍스트 확정
4. **파일 생성** — 확정 후 지정 포맷으로 생성

"최종 출력해줘" → 1~4 한 번에 진행.

### 제목 생성 기준

(1) 핵심 키워드 포함 (2) 15자 이내 권장 (3) 글의 톤과 일치 (4) 독자의 문제 또는 판단 기준이 드러남

### 산출물

| 파일 | 설명 |
|------|------|
| `final.docx` 또는 `final.md` | 최종 완성본 (기본: .docx) |
| `final.html` | 웹 게시용 (선택) |

문서 생성 전 반드시 `/mnt/skills/public/docx/SKILL.md`를 읽고 따른다.

## 스킬 간 연계

### 이 스킬의 범위

단편 콘텐츠(1회 게시 단위)를 처리한다. 범위 밖: 장편(책) → book-writing / 에세이 → write-essay / 강의 슬라이드 → lecture-slides / 브런치 → write-brunch / 페이스북 → write-facebook / 블로그(sshong.com) → write-blog

### 타 스킬 콘텐츠를 입력으로 받을 때

| 입력 | 처리 방식 |
|------|----------|
| 책 챕터 → 칼럼형 | 핵심 논지 1~2개 추출 → 칼럼형 구조로 재구성 |
| 책 챕터 → 문서형 | 서론-본론-결론으로 재배치. 원본의 30~50% |
| 강의 스크립트 → 칼럼형 | 구어체→문어체, 시간순서→논증순서 |
| 에세이 → 칼럼형 | 경험 중심→논증 구조. 관찰→주장→근거→결론 |

### 이 스킬 → 타 스킬

이 스킬은 Step 5까지 완료. 이후 변환은 대상 스킬이 담당.
